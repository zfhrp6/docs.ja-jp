## <a name="choosing-an-authentication-scheme"></a><span data-ttu-id="62e3a-145">認証方式の選択</span><span class="sxs-lookup"><span data-stu-id="62e3a-145">Choosing an Authentication Scheme</span></span>  
 <span data-ttu-id="62e3a-146">HTTP サーバーに使用できる認証方式を選択する際には、以下の項目について検討します。</span><span class="sxs-lookup"><span data-stu-id="62e3a-146">When selecting the potential authentication schemes for an HTTP server, a few items to consider include the following:</span></span>  
  
-   <span data-ttu-id="62e3a-147">リソースを保護する必要があるかどうかを検討します。</span><span class="sxs-lookup"><span data-stu-id="62e3a-147">Consider whether the resource needs to be protected.</span></span> <span data-ttu-id="62e3a-148">HTTP 認証では、より多くのデータを送信する必要があるため、クライアントとの相互運用性が制限される可能性があります。</span><span class="sxs-lookup"><span data-stu-id="62e3a-148">Using HTTP authentication requires transmitting more data and can limit interoperability with clients.</span></span> <span data-ttu-id="62e3a-149">保護する必要のないリソースには、匿名アクセスを許可してください。</span><span class="sxs-lookup"><span data-stu-id="62e3a-149">Allow anonymous access to resources that do not need to be protected.</span></span>  
  
-   <span data-ttu-id="62e3a-150">リソースを保護する必要がある場合は、必要なレベルのセキュリティがどの認証方式によって提供されるかを検討します。</span><span class="sxs-lookup"><span data-stu-id="62e3a-150">If the resource needs to be protected, consider which authentication schemes provide the required level of security.</span></span> <span data-ttu-id="62e3a-151">ここで説明した中で最も弱い標準の認証方式は、基本認証です。</span><span class="sxs-lookup"><span data-stu-id="62e3a-151">The weakest standard authentication scheme discussed here is Basic authentication.</span></span> <span data-ttu-id="62e3a-152">基本認証では、ユーザーの資格情報が保護されません。</span><span class="sxs-lookup"><span data-stu-id="62e3a-152">Basic authentication does not protect the user's credentials.</span></span> <span data-ttu-id="62e3a-153">最も強力な標準の認証方式は、ネゴシエート認証で選択される Kerberos プロトコルです。</span><span class="sxs-lookup"><span data-stu-id="62e3a-153">The strongest standard authentication scheme is Negotiate authentication, resulting in the Kerberos protocol.</span></span>  
  
-   <span data-ttu-id="62e3a-154">受け入れる準備ができていない認証方式、またはリソースを十分にセキュリティで保護できない認証方式をサーバーが (WWW 認証ヘッダー内で) 提示しないことを確認してください。</span><span class="sxs-lookup"><span data-stu-id="62e3a-154">A server should not present (in the WWW-Authentication headers) any scheme that it is not prepared to accept or that does not adequately secure the protected resource.</span></span> <span data-ttu-id="62e3a-155">クライアントは、サーバーによって提示された認証方式を自由に選択します。</span><span class="sxs-lookup"><span data-stu-id="62e3a-155">Clients are free to choose between any of the authentication schemes the server presents.</span></span> <span data-ttu-id="62e3a-156">クライアントによっては、弱い認証方式、またはサーバーのリストにある最初の認証方式を既定で選択します。</span><span class="sxs-lookup"><span data-stu-id="62e3a-156">Some clients default to a weak authentication scheme or the first authentication scheme in the server's list.</span></span>
