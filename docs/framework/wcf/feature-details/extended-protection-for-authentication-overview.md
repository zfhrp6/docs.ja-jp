# <a name="extended-protection-for-authentication-overview"></a><span data-ttu-id="1a4cb-102">認証の拡張保護の概要</span><span class="sxs-lookup"><span data-stu-id="1a4cb-102">Extended Protection for Authentication Overview</span></span>
<span data-ttu-id="1a4cb-103">認証時の拡張保護によって、攻撃者がクライアントの資格情報をインターセプトして特定のサーバーに転送する man-in-the-middle (MITM) 攻撃を防ぐことができます。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-103">Extended Protection for Authentication helps protect against man-in-the-middle (MITM) attacks, in which an attacker intercepts a client’s credentials and forwards them to a server.</span></span>  
  
 <span data-ttu-id="1a4cb-104">クライアント、サーバー、および攻撃者の 3 つの参加要素が存在するシナリオについて考えてみます。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-104">Consider a scenario with three participants: a client, server, and attacker.</span></span> <span data-ttu-id="1a4cb-105">サーバーには `https://server` という URL があり、一方で攻撃者には `https://attacker` という URL があります。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-105">The server has the URL `https://server`, whereas the attacker has the URL `https://attacker`.</span></span> <span data-ttu-id="1a4cb-106">攻撃者は、自身がサーバーであるかのようにしてクライアントが攻撃者にアクセスするように騙します。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-106">The attacker tricks the client into accessing the attacker as if it were the server.</span></span> <span data-ttu-id="1a4cb-107">次に、攻撃者はサーバーに要求を送信します。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-107">The attacker then sends a request to the server.</span></span> <span data-ttu-id="1a4cb-108">攻撃者がセキュリティで保護されたリソースへのアクセスを試みると、サーバーは攻撃者に対して WWW 認証ヘッダーで応答します。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-108">If the attacker is trying to access a secure resource, the server replies to the attacker with a WWW-Authenticate header.</span></span> <span data-ttu-id="1a4cb-109">攻撃者は認証情報を持っていないため、WWW 認証ヘッダーをクライアントに送信します。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-109">The attacker does not have the authentication information, so it sends the WWW-Authenticate header on to the client.</span></span> <span data-ttu-id="1a4cb-110">クライアントは攻撃者に WWW 認証ヘッダーを送信し、攻撃者はそのヘッダーをサーバーに送信してクライアントの資格情報を利用してセキュリティで保護されたリソースにアクセスします。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-110">The client sends the Authorization header to the attacker, and the attacker sends the header on to the server and gets access to the secure resources using the client’s credentials.</span></span>  
  
 <span data-ttu-id="1a4cb-111">現在のところ、クライアント アプリケーションが HTTPS で Kerberos、Digest または NTLM を使用してサーバーに対する認証を実行する場合、最初にトランスポート レベルのセキュリティ (TLS) チャネルが構築され、このチャネルを使用して認証が行われます。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-111">Currently, when a client application authenticates itself to the server using Kerberos, Digest, or NTLM using HTTPS, a Transport Level Security (TLS) channel is first established and authentication takes place using this channel.</span></span> <span data-ttu-id="1a4cb-112">しかし、Secure Sockets Layer (SSL) によって生成されるセッション キーと認証時に生成されるセッション キーはバインドされていません。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-112">However, there is no binding between the session key generated by Secure Sockets Layer (SSL) and the session key that is generated during authentication.</span></span> <span data-ttu-id="1a4cb-113">そのため前述のシナリオでは、TLS (HTTPS チャネルなど) で通信が行われると、2 つの SSL チャネルが作成されます。1 つはクライアントと攻撃者間のチャネルで、もう 1 つは攻撃者とサーバー間のチャネルです。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-113">So, in the previous scenario, if the communication takes places over a TLS (such as an HTTPS channel), there are two SSL channels created: one between the client and the attacker, and another between the attacker and the server.</span></span> <span data-ttu-id="1a4cb-114">まず、クライアントの資格情報がクライアントと攻撃者間の SSL チャネルでクライアントからサーバーに送信され、次に、攻撃者とサーバー間のチャネルでクライアントの資格情報が送信されます。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-114">The client’s credentials are sent from the client to the server first over the SSL channel between the client and the attacker and then over the channel between the attacker and the server.</span></span> <span data-ttu-id="1a4cb-115">クライアントの資格情報がサーバーに届くと、サーバーは情報が送信されたチャネルを検出せずに、クライアントではなく攻撃者から発信された資格情報を確認します。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-115">Once the client’s credentials reach the server, the server verifies the credentials without detecting that the channel over which those credentials were sent originated with the attacker and not the client.</span></span>  
  
 <span data-ttu-id="1a4cb-116">これを解決するには、TLS で保護された外部チャネルとクライアントが認証されている内部チャネルを使用して、チャネル バインディング トークン (CBT) をサーバーに渡します。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-116">The solution is to use a TLS-secured outer channel and a client-authenticated inner channel, and to pass a Channel Binding Token (CBT) to the server.</span></span> <span data-ttu-id="1a4cb-117">CBT は TLS で保護された外部チャネルのプロパティで、外部チャネルをクライアントが認証されている内部チャネルでの対話とバインドするために使用します。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-117">The CBT is a property of the TLS-secured outer channel, and is used to bind the outer channel to a conversation over the client-authenticated inner channel.</span></span>  
  
 <span data-ttu-id="1a4cb-118">前述のシナリオでは、クライアントと攻撃者間の TLS チャネルの CBT は、サーバーに送信される認証情報によって結び付けられます。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-118">In the previous scenario, the CBT of the client-attacker TLS channel is merged with the authorization information that is sent to the server.</span></span> <span data-ttu-id="1a4cb-119">CBT 管理サーバーは、クライアントと攻撃者間のチャネルに対応するクライアント認証情報内の CBT を、攻撃者とサーバー間のチャネルに追加されている CBT と比較します。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-119">A CBT-aware server compares the CBT contained in the client authentication information, which corresponds to the client-attacker channel, to the CBT attached to the attacker-server channel.</span></span> <span data-ttu-id="1a4cb-120">CBT はチャネルの送信先によって特定されるため、クライアントと攻撃者間の CBT は、攻撃者とサーバー間の CBT とは一致しません。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-120">A CBT is specific to a channel’s destination, so the client-attacker CBT does not match the attacker-server CBT.</span></span> <span data-ttu-id="1a4cb-121">これにより、サーバーは MITM 攻撃を検出して認証要求を拒否することができます。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-121">This lets the server detect the MITM attack and refuse the authentication request.</span></span>  
  
 <span data-ttu-id="1a4cb-122">クライアント側が構成設定する必要は一切ありません。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-122">The client side does not require any configuration setting.</span></span> <span data-ttu-id="1a4cb-123">クライアントが CBT をサーバーに渡すように更新しておくと、常にこの動作が実行されます。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-123">Once the client has been updated to pass the CBT to the server, it always does so.</span></span> <span data-ttu-id="1a4cb-124">サーバーを更新しておくと、サーバーが CBT を使用するように設定したり、CBT を無視するように設定することもできます。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-124">If the server has also been updated, it can be configured to use the CBT or ignore it.</span></span> <span data-ttu-id="1a4cb-125">サーバーを更新しない場合、CBT は無視されます。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-125">If it has not been updated, it ignores it.</span></span>  
  
 <span data-ttu-id="1a4cb-126">サーバーには、次の保護レベルを設定できます。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-126">The server can have the following levels of protection:</span></span>  
  
-   <span data-ttu-id="1a4cb-127">なし。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-127">None.</span></span> <span data-ttu-id="1a4cb-128">チャネル バインディングの検証は実行されません。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-128">No channel binding validation is performed.</span></span> <span data-ttu-id="1a4cb-129">更新されていないすべてのサーバーの動作です。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-129">This is the behavior of all servers that have not been updated.</span></span>  
  
-   <span data-ttu-id="1a4cb-130">一部。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-130">Partial.</span></span> <span data-ttu-id="1a4cb-131">更新されたすべてのクライアントは、サーバーにチャネル バインディング情報を提供する必要があります。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-131">All clients that have been updated must provide channel binding information to the server.</span></span> <span data-ttu-id="1a4cb-132">クライアントが更新されていなければ、その必要はありません。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-132">Clients that have not been updated do not have to do so.</span></span> <span data-ttu-id="1a4cb-133">アプリケーションの互換性を許容する中間のオプションです。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-133">This is an intermediate option that allows for application compatibility.</span></span>  
  
-   <span data-ttu-id="1a4cb-134">完全。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-134">Full.</span></span> <span data-ttu-id="1a4cb-135">すべてのクライアントがチャネル バインディング情報を提供する必要があります。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-135">All clients must provide channel binding information.</span></span> <span data-ttu-id="1a4cb-136">サーバーは、クライアントが要求しないクライアントからの認証要求を拒否します。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-136">The server rejects authentication requests from clients that do not do so.</span></span>  
  
 <span data-ttu-id="1a4cb-137">詳細については、Win7 CBT/拡張保護サンプルを参照してください。</span><span class="sxs-lookup"><span data-stu-id="1a4cb-137">For more information, see the Win7 CBT/Extended Protection sample.</span></span>
